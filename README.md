# рҹҢҫ Farm Twin AI

A Digital Twin platform for agriculture вҖ” combining IoT sensor data, weather
inputs, and AI models into a single virtual model of a farm, so decisions
about irrigation, planting, and spending are backed by data instead of guesswork.

## Features

| # | Module | What it does |
|---|--------|---------------|
| 01 | **Yield Prediction** | Estimates future crop production from farm area, soil pH, climate, and fertilizer inputs. |
| 02 | **Disease Risk Prediction** | Flags conditions favorable to fungal/bacterial outbreaks before symptoms appear. |
| 03 | **Smart Irrigation** | Recommends how much (and when) to water, based on soil moisture, temperature, and rainfall forecast. |
| 04 | **Profit Estimation** | Models gross revenue, cost, net profit, margin, and breakeven yield. |
| 05 | **Crop Recommendation** | Matches soil NPK, pH, and climate against crop profiles to suggest the best-fit crop. |

- **Frontend** вҖ” React 18 + Vite, calls the backend's REST API.
- **Backend** вҖ” Spring Boot 3 API gateway. Validates requests, then proxies
  them to the ML microservice via a reactive `WebClient`. This is where
  you'd add auth, persistence, and business logic as the app grows.
- **ML service** вҖ” FastAPI microservice with one endpoint per feature.
  Ships with lightweight, explainable rule-based models so the whole stack
  runs with **zero external datasets or API keys**. Each model file
  (`ml-service/app/models/*.py`) documents exactly how to swap in a trained
  scikit-learn / XGBoost / TensorFlow model without touching the API contract.

## Tech Stack

- **Frontend:** React, Vite, React Router, Axios, Recharts
- **Backend:** Java 17, Spring Boot 3 (Web, WebFlux, Validation, Data JPA), H2
- **ML Service:** Python 3.11, FastAPI, Pydantic
- **Infra:** Docker, Docker Compose (AWS-ready вҖ” see [Deployment](#deployment))

## Quick Start

### Option A вҖ” Docker Compose (recommended)

```bash
git clone https://github.com/<your-username>/farm-twin-ai.git
cd farm-twin-ai
docker compose up --build
```

Then open:
- Frontend вҶ’ http://localhost:3000
- Backend API вҶ’ http://localhost:8080/api/health
- ML service docs (Swagger UI) вҶ’ http://localhost:8000/docs

### Option B вҖ” Run each service locally

**1. ML service**
```bash
cd ml-service
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```

**2. Backend**
```bash
cd backend
mvn spring-boot:run
```
The backend reads `ML_SERVICE_URL` (defaults to `http://localhost:8000`).

**3. Frontend**
```bash
cd frontend
npm install
npm run dev
```
Vite dev server proxies `/api/*` to `http://localhost:8080`, so open
http://localhost:5173.

## API Reference

All backend endpoints are under `/api`:

| Method | Endpoint | Description |
|--------|----------|--------------|
| POST | `/api/yield/predict` | Predict crop yield |
| POST | `/api/disease/risk` | Predict disease risk |
| POST | `/api/irrigation/recommend` | Get irrigation recommendation |
| POST | `/api/profit/estimate` | Estimate profit |
| POST | `/api/crop/recommend` | Recommend best-fit crop |
| GET | `/api/health` | Backend health check |

Example request:

```bash
curl -X POST http://localhost:8080/api/yield/predict \
  -H "Content-Type: application/json" \
  -d '{
    "crop": "wheat",
    "farmAreaHectares": 2.5,
    "soilPh": 6.5,
    "avgTemperatureC": 22,
    "avgRainfallMm": 600,
    "fertilizerKgPerHectare": 60
  }'
```

The ML microservice can also be called directly for testing вҖ” full interactive
docs are auto-generated by FastAPI at `/docs`.

## Roadmap / Where to Extend

- Swap rule-based models in `ml-service/app/models/` for trained models
  (each file documents the expected input/output shape).
- Add persistence: wire up farm profiles, historical sensor readings, and
  prediction history using the JPA setup already scaffolded in the backend.
- Add real IoT ingestion (MQTT/HTTP) and weather API integration feeding
  into the ML service inputs.
- Add authentication (Spring Security + JWT) once multiple farmers/tenants
  are supported.

## Deployment

The `docker-compose.yml` is a starting point for local development. For AWS:
- Push each service's image to **ECR**.
- Run them on **ECS Fargate** (or EKS) вҖ” three separate services/tasks
  mirroring the compose file.
- Put the frontend behind **CloudFront + S3** (static build) or serve via
  the included Nginx container.
- Replace H2 with **RDS (Postgres)** for the backend's persistence layer.
